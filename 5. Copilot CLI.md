# End-to-End Guided Practice: GitHub Copilot CLI in Codespaces

_This guide provides a hands‑on walkthrough for using the interactive **GitHub Copilot CLI** (`github‑copilot‑cli` or the newer `gh copilot` extension) directly within a browser‑based environment._

> We will use a blank GitHub Codespace, which comes pre‑configured with the necessary authentication and tools, allowing you to focus entirely on learning the interactive agent's capabilities for development, containerization, and system administration.

---

## Phase 1: Environment Setup

We will start by provisioning a fresh, isolated workspace.

- **Navigate to GitHub Codespaces** – Open your browser and go to <https://github.com/codespaces>.
- **Launch a Blank Template** – Click on the *Blank* template under *Explore quick start templates*.
- **Access the Terminal** – Once the editor loads, open the integrated terminal (usually visible at the bottom, or via **Terminal > New Terminal**).

---

## Phase 2: Core Development Workflow

In this phase we will write code, generate test data, and build test suites entirely via conversational commands.

### Step 1: Enter Interactive Mode

**Input** (type in the terminal):

```bash
copilot
```

> *(Note: depending on the exact pre‑installed version, the command might be `gh copilot suggest` or `gh copilot explain`. For this guide we assume the interactive Copilot shell is active.)*

**Expected Output** – The terminal prompt changes, indicating the Copilot agent is actively listening for your instructions.

### Step 2: Scaffold a Program File

**Input**:

```
Write a Python script named `log_parser.py` that reads a log file, counts the number of "ERROR" and "WARNING" lines, and prints the summary. Use standard libraries only.
```

**Expected Output** – Copilot generates the Python code and offers to create the file (e.g., via `echo "..." > log_parser.py`). Accepting the suggestion creates the file in your workspace.

### Step 3: Create Mock Data

**Input**:

```
Create a sample log file named `server.log` with 10 lines of INFO, WARNING, and ERROR messages.
```

**Expected Output** – Copilot generates shell commands to create `server.log` populated with sample text. Accepting the execution creates the file.

### Step 4: Generate Test Cases

**Input**:

```
Create a test file named `test_log_parser.py` using the `unittest` framework to verify `log_parser.py`. Include tests for empty files and files with mixed log levels.
```

**Expected Output** – Copilot analyses `log_parser.py` and generates a corresponding Python test script, then prompts you to create the file.

### Step 5: Explain the Implementation

**Input**:

```
/explain "How does the log_parser.py script handle file reading efficiently without loading massive logs into memory?"
```

**Expected Output** – The agent provides a conversational explanation, highlighting parts of the generated code (e.g., iterating line‑by‑line with a `for` loop) that ensure memory efficiency.

### Step 6: Execute the Tests

**Input** – Exit the interactive mode (`exit` or `Ctrl+C`) and run:

```bash
python3 -m unittest test_log_parser.py
```

**Expected Output** – Standard Python `unittest` output showing tests passing against the logic you built with Copilot.

## Phase 3: Infrastructure and Operations

Drop back into the interactive shell (`copilot`) to practice system‑administration and infrastructure workflows.

### Step 7: Containerize the Application

**Input**:

```
Write a Dockerfile for `log_parser.py`. Use a lightweight Alpine Linux Python base image, copy the script, and set the default command to run the script. Ensure it follows best practices for layer caching.
```

**Expected Output** – Copilot generates a Dockerfile using a `python:alpine` base image, orders commands logically (e.g., copying requirements first if any), and sets `CMD` or `ENTRYPOINT`.

### Step 8: Kubernetes Manifest Generation

**Input**:

```
Create a Kubernetes deployment manifest named `deployment.yaml`. It should deploy 3 replicas of a container named "log-parser", requesting 256Mi of memory and 200m CPU, with a rolling update strategy.
```

**Expected Output** – Copilot generates a valid YAML file containing a `Deployment` object with the requested replica count, resource requests, and update strategy.

### Step 9: Complex Cloud CLI Queries

**Input**:

```
Write an AWS CLI command to list all EC2 instances in the `ap-south-1` region that have a tag "Environment" set to "Production", and format the output as a table showing only the InstanceID and State.
```

**Expected Output** – Copilot outputs the exact command:

```bash
aws ec2 describe-instances \
  --region ap-south-1 \
  --filters "Name=tag:Environment,Values=Production" \
  --query "Reservations[*].Instances[*].[InstanceId,State.Name]" \
  --output table
```

### Step 10: Advanced Text‑Processing Pipelines

**Input**:

```
Write a bash one‑liner using `awk` to parse `server.log`, count the total number of occurrences for each log level (INFO, WARNING, ERROR), and sort them in descending order.
```

**Expected Output** – Copilot generates a pipeline such as:

```bash
awk '{print $1}' server.log | sort | uniq -c | sort -nr
```

It will offer to execute this pipeline directly against the `server.log` file created earlier.

---

## Phase 4: Cleanup

Once you have finished practicing, it is good practice to shut down and delete the Codespace to avoid consuming active usage hours or storage quota.

### Step 11: Exit the CLI and Delete the Workspace

**Input** – If you are still in the interactive Copilot session, type `exit` or press `Ctrl+C` to return to the standard Bash prompt.

**Action**:

1. Close the browser tab containing your Codespace editor.
2. Navigate back to your Codespaces dashboard: <https://github.com/codespaces>.
3. Locate the blank Codespace you just used in the list of active environments.
4. Click the three‑dot menu (options) on the right side of that entry.
5. Select **Delete**.

**Expected Output** – The Codespace is destroyed, freeing up your cloud resources and storage footprint.
